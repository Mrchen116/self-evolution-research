Hermes Agent · Skill / Memory 自动生成更新机制

研究空间：self-evolution/hermes-agent · 最后校对 2026-05-18 · 右上角可切换 prompt 的中英文显示

1 · 三层结构总览

Hermes 让 skill 和 memory 自动生成 / 更新的整套机制由三层并行构成。理解这三层是其它细节的前提：

① 主 Agent

在每个 turn 里和用户对话，可以主动调用 memory / skill_manage 工具。System prompt 里有 MEMORY_GUIDANCE / SKILLS_GUIDANCE 常驻提醒它"该记就记、该改就改"。

触发：每个 turn 都跑

② Per-turn Review Fork

当主 agent 一段时间没主动管 memory / skill 时，turn 结束后在后台 daemon thread 里 fork 一个新 agent，让它专门审视："这段对话里有没有该沉淀的东西？"

触发：每 ~10 turn / 10 tool iteration

③ Curator

跨 session 的长期管家。用 auxiliary 模型整理整个 skill 库：把长期不用的 archive、把多个主题相近的窄 skill 合并成一个宽口径的 umbrella（"伞"——能涵盖一整类问题的通用 skill，对立面是为某次任务现做的窄 skill）、把过时的 patch 掉。只动 agent-created skill。

触发：CLI 启动 / gateway 运行时按 schedule 跑

三层互不阻塞主对话：①是主线；②跑在 daemon thread，turn 结束才启动；③用独立 auxiliary client，从不碰主 session 的 prompt cache。

2 · 触发机制：两个计数器

Hermes 用两个非常朴素的计数器来决定 per-turn review 是否启动。先明确两个概念：

User Turn（用户轮次）

用户发一条消息 → agent 给出最终回复，算 1 个 turn。即使期间调了 5 个工具，仍然只是 1 个 turn。

Tool Iteration（工具迭代）

主 agent 内部一次"思考 → 调用工具 → 拿到结果"的循环。1 个 turn 通常包含多个 iteration。

类别计数器+1 时机归零时机默认阈值
Memory _turns_since_memory 每个 user turn 结束 agent 调用了 memory 工具 10 turns
Skill _iters_since_skill 每个 tool iteration 进入 agent 调用了 skill_manage 工具 10 iterations

用人话说："agent 已经做了很多事但一直没主动整理" 时，系统在当前 turn 结束后自动派 review fork 去审视。如果 agent 自己经常主动调 memory / skill_manage，计数器就会反复归零，review 几乎不会触发。

3 · 一个 turn 的完整时序

关键点：review fork 在 turn 结束后才启动，不阻塞主对话。用户看到主 agent 的回复时，review 才刚开始。

用户实际看到什么

整套机制对用户的可见 UI 只有一行——review fork 完成且实际写了东西时，在终端打印：

  💾 Self-improvement review: Skill 'pytest-xdist-debug' created
  💾 Self-improvement review: Memory updated · User profile updated

没有动作就完全静默（fork 在 daemon thread 里悄悄死掉）。即用户大多数 turn 根本察觉不到 review 存在。

传给 fork 的 messages_snapshot 里装了什么

list(messages) 的一份浅拷贝——完整对话历史，包含本次 turn 里的所有：

  • user 消息（含 Honcho 注入的画像背景，如果有）
  • assistant 消息（含 tool_calls 字段）
  • tool result 消息（agent 调每个工具拿到的返回）
  • system reminders（如果有）

fork 启动时把这份快照作为 conversation_history 参数传入新 AIAgent，再把 review prompt 作为新的 user message——这样 fork 看到的"对话"是【完整真实历史 + 一条 review 任务】，可以基于真实证据决策。

4 · Review Fork 是什么

主 agent 不能"暂停去想要不要存 skill"，所以 Hermes 选了一种更聪明的做法：把自己 fork 一份，让 fork 去做 review，fork 跑完自然死亡。

"fork" 不是 os.fork()。这里指在 threading.Thread(daemon=True) 里实例化一个全新的 AIAgent Python 对象，把主 agent 的部分状态（model、provider、credentials、cached system prompt、session_id 等）作为构造参数传过去。fork 和主 agent 跑在同一个进程的不同线程，共享解释器但不共享对话上下文。

Fork 的 9 项关键配置（agent/background_review.py:381-450）

配置作用
model / provider与主 agent 完全一致（除非主 agent 用 codex_app_server，则 downgrade 到 codex_responses）
max_iterations=16review 限定最多 16 轮思考
quiet_mode=True不在终端打日常输出

5 · Per-turn Review 的三段 Prompt

fork 启动时会根据触发情况，注入三个 prompt 之一作为 user_message。这三段 prompt 是整个 self-improvement 机制的"灵魂"——没有任何代码逻辑去做 diff、对比、规则匹配，所有"该不该记 / 该不该改 / 怎么改"的决策全靠这三段自然语言。

② Skill Review Prompt（约 95 行，全文）

这段 prompt 是整个 skill 体系的"宪法"。结构概览：

结构内容
开篇基调"Be ACTIVE — most sessions produce at least one skill update"
目标形态追求 class-level umbrella skill + references/；反对扁平窄 skill 列表（class-level = "类级"：命名落在一整类问题上，如 pull-requests、postgres-migrations；对立面是 instance-level 的一次性 skill，如 fix-build-error-2026-04-12。class-level 和 umbrella 基本是同义词，前者强调命名粒度，后者强调"把相关条目收纳在一把伞下"的结构）
触发信号用户纠正 style / format / workflow；non-trivial 技术；已加载 skill 被证明 wrong
4 级偏好顺序patch loaded → patch umbrella → write_file → create new umbrella
Do NOT capture环境依赖错误 / 工具负面断言 / 临时 transient error / one-off narrative
结尾"'Nothing to save.' is real but NOT the default"
 
Review the conversation above and update the skill library. Be ACTIVE —
most sessions produce at least one skill update, even if small. A pass that
does nothing is a missed learning opportunity, not a neutral outcome.

Target shape of the library: CLASS-LEVEL skills, each with a rich SKILL.md
and a `references/` directory for session-specific detail. Not a long flat
list of narrow one-session-one-skill entries. This shapes HOW you update,
not WHETHER you update.

Signals to look for (any one of these warrants action):
  • User corrected your style, tone, format, legibility, or verbosity.
    Frustration signals like 'stop doing X', 'this is too verbose', 'don't
    format like this', 'why are you explaining', 'just give me the answer',
    'you always do Y and I hate it', or an explicit 'remember this' are
    FIRST-CLASS skill signals, not just memory signals. Update the relevant
    skill(s) to embed the preference so the next session starts already
    knowing.
  • User corrected your workflow, approach, or sequence of steps. Encode
    the correction as a pitfall or explicit step in the skill that governs
    that class of task.
  • Non-trivial technique, fix, workaround, debugging path, or tool-usage
    pattern emerged that a future session would benefit from. Capture it.
  • A skill that got loaded or consulted this session turned out to be
    wrong, missing a step, or outdated. Patch it NOW.

Preference order — prefer the earliest action that fits, but do pick one
when a signal above fired:
  1. UPDATE A CURRENTLY-LOADED SKILL. Look back through the conversation
     for skills the user loaded via /skill-name or you read via skill_view.
     If any of them covers the territory of the new learning, PATCH that
     one first. It is the skill that was in play, so it's the right one to
     extend.
  2. UPDATE AN EXISTING UMBRELLA (via skills_list + skill_view). If no
     loaded skill fits but an existing class-level skill does, patch it.
     Add a subsection, a pitfall, or broaden a trigger.
  3. ADD A SUPPORT FILE under an existing umbrella. Skills can be packaged
     with three kinds of support files — use the right directory per kind:
     • `references/.md` — session-specific detail (error transcripts,
       reproduction recipes, provider quirks) AND condensed knowledge banks:
       quoted research, API docs, external authoritative excerpts, or domain
       notes you found while working on the problem. Write it concise and
       for the value of the task, not as a full mirror of upstream docs.
     • `templates/.` — starter files meant to be copied and
       modified (boilerplate configs, scaffolding, a known-good example
       the agent can `reproduce with modifications`).
     • `scripts/.` — statically re-runnable actions the skill
       can invoke directly (verification scripts, fixture generators,
       deterministic probes, anything the agent should run rather than
       hand-type each time).
     Add support files via skill_manage action=write_file with file_path
     starting 'references/', 'templates/', or 'scripts/'. The umbrella's
     SKILL.md should gain a one-line pointer to any new support file so
     future agents know it exists.
  4. CREATE A NEW CLASS-LEVEL UMBRELLA SKILL when no existing skill covers
     the class. The name MUST be at the class level. The name MUST NOT be
     a specific PR number, error string, feature codename, library-alone
     name, or 'fix-X / debug-Y / audit-Z-today' session artifact. If the
     proposed name only makes sense for today's task, it's wrong — fall
     back to (1), (2), or (3).

User-preference embedding (important): when the user expressed a
style/format/workflow preference, the update belongs in the SKILL.md body,
not just in memory. Memory captures 'who the user is and what the current
situation and state of your operations are'; skills capture 'how to do this
class of task for this user'. When they complain about how you handled a
task, the skill that governs that task needs to carry the lesson.

If you notice two existing skills that overlap, note it in your reply —
the background curator handles consolidation at scale.

Do NOT capture (these become persistent self-imposed constraints that bite
you later when the environment changes):
  • Environment-dependent failures: missing binaries, fresh-install errors,
    post-migration path mismatches, 'command not found', unconfigured
    credentials, uninstalled packages. The user can fix these — they are
    not durable rules.
  • Negative claims about tools or features ('browser tools do not work',
    'X tool is broken', 'cannot use Y from execute_code'). These harden
    into refusals the agent cites against itself for months after the
    actual problem was fixed.
  • Session-specific transient errors that resolved before the conversation
    ended. If retrying worked, the lesson is the retry pattern, not the
    original failure.
  • One-off task narratives. A user asking 'summarize today's market' or
    'analyze this PR' is not a class of work that warrants a skill.

If a tool failed because of setup state, capture the FIX (install command,
config step, env var to set) under an existing setup or troubleshooting
skill — never 'this tool does not work' as a standalone constraint.

'Nothing to save.' is a real option but should NOT be the default. If the
session ran smoothly with no corrections and produced no new technique,
just say 'Nothing to save.' and stop. Otherwise, act.
回顾上面的对话，更新 skill 库。要 ACTIVE（主动）——
绝大多数 session 都应该产出至少一次 skill 更新，哪怕很小。
什么都不做的一轮 review 是错过的学习机会，不是中立结果。

目标形态：CLASS-LEVEL（类级）skill，每个都有内容丰富的 SKILL.md
和用于存放 session 细节的 `references/` 子目录。不要做成一长串
"一个 session 对应一个窄 skill" 的扁平列表。这个原则决定的是
"怎么更新"，不是"要不要更新"。

触发信号（任一即应行动）：
  • 用户纠正了你的风格 / 语气 / 格式 / 可读性 / 啰嗦程度。
    带挫败感的信号——'stop doing X' / '这太冗长了' / '别用这种格式' /
    '为什么要解释' / '直接给答案' / '你老是 Y 我特别烦' / 明确说
    'remember this'——都是 FIRST-CLASS 的 skill 信号，不只是 memory
    信号。更新相关 skill，把偏好 embed 进去，下一个 session 就已经知道。
  • 用户纠正了你的 workflow / 做法 / 步骤顺序。把这次纠正编码进
    管这类任务的 skill，写成 pitfall 或显式步骤。
  • 出现了未来 session 能用的 non-trivial 技术 / 修复 / 绕路 /
    debug 路径 / 工具使用模式。捕获下来。
  • 本 session 加载或查过的某个 skill 被证明 wrong / missing a step /
    outdated。立刻 patch。

偏好顺序——优先选最早能 fit 的动作，但只要触发信号出现就必须挑一个：
  1. UPDATE A CURRENTLY-LOADED SKILL（修改本对话已加载的 skill）。
     回看对话里用户通过 /skill-name 加载、或你用 skill_view 读过的
     skill。如果其中哪个覆盖这次的新经验，优先 PATCH 它——它在场，
     就是该扩展的那个。
  2. UPDATE AN EXISTING UMBRELLA（修改已有 umbrella skill）。
     如果没有已加载的 skill 合适，但有一个 class-level 的 skill 合适，
     就 patch 那个。加一个子小节、一条 pitfall、或拓宽一个 trigger。
     用 skills_list + skill_view 找到它。
  3. ADD A SUPPORT FILE under an existing umbrella（往 umbrella 下加
     支持文件）。skill 可以打包三种支持文件，按种类放到对应目录：
     • `references/.md` —— session 特定细节（error 记录、复现
       步骤、provider 怪异行为）以及精炼的知识库（引用的 research、
       API 文档片段、外部权威节录、领域笔记）。要简洁、贴任务，不要
       做成 upstream 文档的完整镜像。
     • `templates/.` —— 用于复制改造的起步文件（boilerplate
       config、脚手架、"复制再改"的已知好示例）。
     • `scripts/.` —— skill 可以直接调用的静态可重跑动作
       （验证脚本、fixture 生成器、确定性探测——任何不希望 agent 每次
       手敲的东西）。
     用 skill_manage action=write_file 加，file_path 以 'references/' /
     'templates/' / 'scripts/' 开头。umbrella 的 SKILL.md 应该加一行
     指向新支持文件的指针，让未来的 agent 知道它存在。
  4. CREATE A NEW CLASS-LEVEL UMBRELLA SKILL（新建 class-level umbrella）
     ——仅当没有任何已有 skill 覆盖这一类。名字必须在类级别。
     绝对不要用 PR 号、特定 error 字符串、feature 代号、单独的库名、
     或 'fix-X / debug-Y / audit-Z-today' 这种 session artifact 当名字。
     如果名字只对今天的任务有意义，就是错的——回到 (1)(2)(3)。

User-preference embedding（重要）：用户表达 style / format / workflow
偏好时，更新属于 SKILL.md body，不只是 memory。memory 记的是"用户是谁、
当前对话和操作状态如何"；skill 记的是"如何为这位用户做这类任务"。
当用户对你处理某类任务的方式表达不满时，管这类任务的 skill 必须承载这条
教训。

如果你发现两个已存在 skill 有 overlap，在回复里提一句——后台 curator
负责大规模整合。

不要 capture（这些会硬化成持久的自我约束，环境一变就咬到自己）：
  • 环境依赖的失败：缺二进制、fresh-install 报错、migration 后路径不一致、
    'command not found'、未配 credential、未装包。用户自己能修——
    这不是 durable 规则。
  • 关于工具或功能的负面断言（'browser tools 不能用' / 'X 工具坏了' /
    'execute_code 里调不到 Y'）。这种会硬化成 agent 反复对自己讲"不能用"
    的拒绝，且会在真正的问题修复几个月之后还继续生效。
  • Session 内 transient error，对话结束前自己 resolve 了。如果重试成了，
    教训是 retry pattern，不是最初的 failure。
  • One-off 任务叙事。用户问 '总结今天的市场' 或 '分析这个 PR' 不构成
    一类工作。

如果某次工具因 setup 状态失败，捕获的应该是 FIX（install 命令、config
步骤、env var）——写到现有的 setup / troubleshooting skill 里——
永远不要写成 '这工具不能用' 这种独立约束。

'Nothing to save.' 是真选项，但不应该是默认。如果 session 顺利、无纠正、
也没产出新技术，就说 'Nothing to save.' 然后停止。否则——行动。
③ Combined Review Prompt（memory + skill 同时）

同骨架的更长版本，把 memory + skill 两条线并到一份 prompt 里，要求 fork 同时审视用户画像信息 + 是否有值得沉淀的经验。结构和 ② 完全一致，只是开头多一段 memory 引导。位置：background_review.py:141-215。

6 · skill_manage 工具的 6 个 action

无论是主 agent 主动写 skill，还是 review fork 自动整理 skill，还是 curator 在长周期做整合，最终都通过同一个工具 skill_manage（tools/skill_manager_tool.py:713-790）：

action作用关键参数
create创建新 skill，写入 ~/.hermes/skills/<name>/SKILL.mdname, content（完整 YAML frontmatter + body）, 可选 category
edit完全重写某个 skill 的 SKILL.mdname, content（完整新版本）
patch对 SKILL.md 或任意支持文件做 fuzzy find-and-replacename, old_string, new_string, 可选 file_path / replace_all
delete"删除" skill（实际是移到 .archive/，可恢复）name, 可选 absorbed_into
write_file在 skill 目录下写一份支持文件name, file_path（必须以 references/ / templates/ / scripts/ 开头）, file_content
remove_file移除支持文件（删空后自动 rmdir 父目录）name, file_path

每次成功后做的三件事

  1. 清空 skill system prompt cache，下一轮立刻看到更新
  2. Provenance 打标：create 时若处在 review fork 内（is_background_review()），标记为 agent-created；前台用户主动 create 的不打标
  3. Telemetry 计数：patch/edit/write_file/remove_file 时 bump_patch(name) 累加修订次数；delete 时 forget(name)

第 2 步的 provenance 是 Curator 工作的前提——Curator 只动 agent-created 的 skill，永远不会改用户主动创建的 skill。

Skill 的三种来源，只有一种被 Curator 管

本机 ~/.hermes/skills/ 下的 skill 实际上来自三条路径，curator 只动其中第三种：

来源判定方式Curator 会动吗？
① Bundled（hermes 自带 / 内置） 名字出现在 .bundled_manifest 里 不会
② Hub-installed（从 skills-hub 装的） 名字出现在 hub 安装记录里 不会
③ Agent-created（review fork 自己写出来的） .usage.json 里这个 skill 的 record 含 curator_managed: true 会

注意：用户在前台主动调 skill_manage(action="create") 创建的 skill 不归任何一类——它既不在 bundled / hub 名单里，.usage.json 里也没被 mark_agent_created() 打标。结果是 curator 一律视为"用户主动作品"，永远不动。

怎么打"agent-created"这个标？

用 Python 的 contextvars.ContextVar（tools/skill_provenance.py）：

  1. review fork 启动时，进入 background_review_context()，把 ContextVar 设为 True
  2. fork 在这个上下文里调 skill_manage(action="create", ...)
  3. dispatcher 成功写入后，调 is_background_review() 查 ContextVar；为 True 就 mark_agent_created(name) 写进 .usage.json
  4. fork 退出，ContextVar 恢复

主 agent 的前台调用拿不到这个 ContextVar = True 的上下文，所以前台 create 不会打标——边界就在这里。

7 · 目录结构与持久化

所有 agent 自己生成 / 更新的 skill 和 memory 都落到本机 ~/.hermes/ 下：

~/.hermes/
├── skills/ # skill 总目录
│ ├── .curator_state # curator 调度状态 (JSON)
│ ├── .archive/ # 被 archive 的 skill（curator 软删除）
│ │ └── <old-skill>/ ...
│ │
│ ├── <skill-name>/ # 一个 umbrella skill
│ │ ├── SKILL.md # YAML frontmatter + Markdown body（必需）
│ │ ├── references/ # session 细节 / 知识库片段
│ │ │ ├── error-transcript-2026-04-12.md
│ │ │ └── api-quirks.md
│ │ ├── templates/ # 可复制改造的起步文件
│ │ │ └── starter.toml
│ │ └── scripts/ # 可静态重跑的脚本
│ │ └── verify.py
│ │
│ └── <category>/ # 可选分类层，仅影响目录组织
│ └── <skill-name>/
│ └── SKILL.md
│
├── memories/ # memory 总目录

关于 category

9 · Curator：跨 session 的长期管家

这是本文档最长的一节——因为 Curator 是独立的第二条 skill 进化路径，和前面 1-8 节讲的 per-turn review 完全不同。Per-turn review 解决的是"当下这段对话有没有该记的"，curator 解决的是"整个 skill 库是不是越来越乱"。

"session" 在 Hermes 里指什么

一次 hermes CLI 从启动到退出 = 1 个 session；或 gateway 模式下一次 web 连接 = 1 个 session。memory（MEMORY.md / USER.md）跨 session 持久；conversation history 按 session 隔离存到 SQLite，可通过 session_search 跨 session 检索。

"auxiliary model" 是什么

Hermes 在配置里允许声明 auxiliary.<task>.{provider,model,api_key,base_url}，用于把一些"体力活"路由到比主对话模型便宜的模型上。Curator 用 auxiliary.curator.*，session_search 用 auxiliary.session_search.*。可以是同一家厂商的 Haiku/Mini，也可以是另一家（Gemini Flash 等）；不配的话 fall back 到主对话模型。

9.1 为什么需要 Curator？

per-turn review fork 每天都在生 skill。三个月后，会出现这些问题：

问题 ① 窄而碎

出现一堆名字像 fix-build-error-2026-04-12 的 skill——当初某个 session 临时创建的，根本不是 class-level。

问题 ② 重复堆叠

pr-review-checklist / pr-triage / pr-salvage 三个 skill 主题高度重叠，应该 merge 成一个 umbrella。

问题 ③ 长期不用

某个 skill 几个月没被 view 也没被 patch——也许早就过时了，但它还在 system prompt 索引里占位。

问题 ④ 失效但未察觉

某个 skill 当初有效，但所引用的工具 / API 已经变了。需要有人定期巡检。

Curator 就是定期巡检 + 整合的"图书管理员"。

9.2 触发条件：不是 cron，是 inactivity-triggered

Curator 没有独立的后台 daemon 在跑。它的入口是 maybe_run_curator()（agent/curator.py:1763），在两个时机被显式调用：

时机调用方idle 假设
CLI 启动时 cli.py:11702 在用户启动 hermes 命令行时尝试一次 idle_for_seconds=float("inf")（用户刚启动 = 完全 idle）
Gateway 运行时 gateway/run.py:16780 在 web gateway 长期运行时按间隔检查 由 gateway 测量真实 idle 时长
手动 hermes curator run / hermes curator run --dry 用户主动触发，绕过 idle gate

maybe_run_curator() 内部有两道闸门：

def maybe_run_curator(*, idle_for_seconds, on_summary):
    if not should_run_now():           # 闸门 ①：距上次跑足够久吗？
        return None
    if idle_for_seconds is not None:
        if idle_for_seconds < min_idle_s:  # 闸门 ②：agent 真的空闲吗？
            return None
    return run_curator_review(on_summary=on_summary)

两道闸门都过了才真正启动。失败始终 swallowed（except Exception: return None），永远不会破坏 session 启动。

9.4 Skill 生命周期状态机

每个 agent-created skill 在 tools/skill_usage.py 里都有 state 字段，三个值：

状态转换的 anchor 是 last_activity_at，定义为 last_used_at / last_viewed_at / last_patched_at 三者取最新（tools/skill_usage.py:120）。所以下列任一动作都会刷新 anchor、把 stale 拉回 active：

  • 主 agent 或 fork 调 skill_view(name="X") 读这个 skill → bump_view
  • 用户在 CLI 里输 /X（加载 skill 进对话）→ bump_use
  • agent 或 review fork 调 skill_manage(action="patch"/"edit"/"write_file"/"remove_file", name="X") → bump_patch

"create 时间"不算 activity——刚生出来的 skill 如果没人碰，到了 90 天阈值照样会被 archive。状态转换逻辑（curator.py:282-294）：

  • anchor ≤ 90d 前 且当前不是 archived → archive（移到 .archive/）
  • anchor ≤ 30d 前 且当前是 active → 标 stale
  • anchor > 30d 内 且当前是 stale → reactivate（恢复 active）

9.7 Curator 的 LLM Prompt（重点）

CURATOR_REVIEW_PROMPT（curator.py:330-445）和 per-turn review 的 skill prompt 完全不同。后者是"看这次对话有没有该存的"，前者是 "看整个库该怎么 umbrella 化"。这段 prompt 极度强势，明确指令 LLM 必须 consolidate，"不动"不是中立选项。

CURATOR_REVIEW_PROMPT 全文 
You are running as Hermes' background skill CURATOR. This is an
UMBRELLA-BUILDING consolidation pass, not a passive audit and not a
duplicate-finder.

The goal of the skill collection is a LIBRARY OF CLASS-LEVEL
INSTRUCTIONS AND EXPERIENTIAL KNOWLEDGE. A collection of hundreds of
narrow skills where each one captures one session's specific bug is
a FAILURE of the library — not a feature. An agent searching skills
matches on descriptions, not on exact names; one broad umbrella
skill with labeled subsections beats five narrow siblings for
discoverability, not the other way around.

The right target shape is CLASS-LEVEL skills with rich SKILL.md
bodies + `references/`, `templates/`, and `scripts/` subfiles for
session-specific detail — not one-session-one-skill micro-entries.

Hard rules — do not violate:
1. DO NOT touch bundled or hub-installed skills. The candidate list
   below is already filtered to agent-created skills only.
2. DO NOT delete any skill. Archiving (moving the skill's directory
   into ~/.hermes/skills/.archive/) is the maximum destructive action.
   Archives are recoverable; deletion is not.
3. DO NOT touch skills shown as pinned=yes. Skip them entirely.
4. DO NOT use usage counters as a reason to skip consolidation. The
   counters are new and often mostly zero. Judge overlap on CONTENT,
   not on use_count. 'use=0' is not evidence a skill is valuable; it's
   absence of evidence either way.
5. DO NOT reject consolidation on the grounds that 'each skill has
   a distinct trigger'. Pairwise distinctness is the wrong bar. The
   right bar is: 'would a human maintainer write this as N separate
   skills, or as one skill with N labeled subsections?' When the
   answer is the latter, merge.

How to work — not optional:
1. Scan the full candidate list. Identify PREFIX CLUSTERS (skills
   sharing a first word or domain keyword). Examples you are likely
   to find: hermes-config-*, hermes-dashboard-*, gateway-*, codex-*,
   ollama-*, anthropic-*, gemini-*, mcp-*, salvage-*, pr-*,
   competitor-*, python-*, security-*, etc. Expect 10-25 clusters.
2. For each cluster with 2+ members, do NOT ask 'are these pairs
   overlapping?' — ask 'what is the UMBRELLA CLASS these skills all
   serve? Would a maintainer name that class and write one skill for
   it?' If yes, pick (or create) the umbrella and absorb the siblings
   into it.
3. Three ways to consolidate — use the right one per cluster:
   a. MERGE INTO EXISTING UMBRELLA — one skill in the cluster is
      already broad enough to be the umbrella. Patch it to add a
      labeled section for each sibling's unique insight, then archive
      the siblings.
   b. CREATE A NEW UMBRELLA SKILL.md — no existing member is broad
      enough. Use skill_manage action=create to write a new class-
      level skill whose SKILL.md covers the shared workflow and has
      short labeled subsections. Archive the now-absorbed narrow
      siblings.
   c. DEMOTE TO REFERENCES/TEMPLATES/SCRIPTS — a sibling has
      narrow-but-valuable session-specific content. Move it into the
      umbrella's appropriate support directory:
      • `references/.md` for session-specific detail OR
        condensed knowledge banks (quoted research, API docs excerpts,
        domain notes, provider quirks, reproduction recipes)
      • `templates/.` for starter files meant to be
        copied and modified
      • `scripts/.` for statically re-runnable actions
        (verification scripts, fixture generators, probes)
      Then archive the old sibling.
4. Also flag skills whose NAME is too narrow (contains a PR number,
   a feature codename, a specific error string, an 'audit' /
   'diagnosis' / 'salvage' session artifact). These almost always
   belong as a subsection or support file under a class-level umbrella.
5. Iterate. After one consolidation round, scan the remaining set
   and look for the NEXT umbrella opportunity. Don't stop after 3
   merges.

Your toolset:
  - skills_list, skill_view        — read the current landscape
  - skill_manage action=patch      — add sections to the umbrella
  - skill_manage action=create     — create a new umbrella SKILL.md
  - skill_manage action=write_file — add a references/, templates/,
    or scripts/ file under an existing skill

    `absorbed_into=` when you've merged its content into
    another skill, or `absorbed_into=""` when you're truly pruning
    with no forwarding target.
  - terminal                       — mv a sibling into the archive
    OR move its content into a support subfile

'keep' is a legitimate decision ONLY when the skill is already a
class-level umbrella and none of the proposed merges would improve
discoverability. 'This is narrow but distinct from its siblings'
is NOT a reason to keep — it's a reason to move it under an
umbrella as a subsection or support file.

Expected output: real umbrella-ification. Process every obvious

stopped too early — go back and look at the clusters you left

      • `references/.md` —— session 特定细节 或 精炼知识库
        （研究引用、API 文档节录、领域笔记、provider 怪异行为、复现配方）
      • `templates/.` —— 用于复制改造的起步文件
      • `scripts/.` —— 静态可重跑的动作（验证脚本、
        fixture 生成器、探测）
      然后 archive 老的 sibling。
4. 同时标出 NAME 太窄的 skill（含 PR 号、feature 代号、特定 error 字符串、
   或 'audit' / 'diagnosis' / 'salvage' 这类 session artifact）。
   这些几乎总是该作为某个 class-level umbrella 的子小节或支持文件。
5. 迭代。第一轮整合做完后，扫剩下的 skill，找下一个 umbrella 机会。
   不要 merge 3 个就停。

你的工具集：
  - skills_list, skill_view        —— 读当前 skill 全景
  - skill_manage action=patch      —— 给 umbrella 加小节
  - skill_manage action=create     —— 新建 umbrella SKILL.md
  - skill_manage action=write_file —— 在某个已存在 skill 下加
    references/ / templates/ / scripts/ 文件
  - skill_manage action=delete     —— archive 一个 skill。**必须**传
    `absorbed_into=` 当你已经把它的内容合并到另一个 skill 里；
    或传 `absorbed_into=""` 当你是真的在 prune、没有 forwarding target。
    这个字段驱动 cron-job 的 skill 引用迁移——事后从你 YAML summary 里
    猜的话很 fragile。
  - terminal                       —— mv sibling 进 archive 或把它的
    内容移进 support 子文件

'keep'（保留不动）是合法决定 **只有**当：这个 skill 已经是 class-level
umbrella，且没有任何提议中的 merge 能改善可发现性。
"它窄，但和 sibling 不同" 不是 keep 的理由——这是把它移到某个 umbrella
下做子小节或支持文件的理由。

期望输出：真正的 umbrella 化。处理每一个明显的 cluster。如果你以
< 10 个 archive 结束这一轮，说明你停得太早——回去看你放过的 cluster。

完成后，写一段人类可读 summary，**以及**一段结构化机器可读 block，
让下游工具能区分 consolidation 和 pruning。格式严格照抄：

## Structured summary (required)
```yaml
consolidations:
  - from: <old-skill-name>
    into: <umbrella-skill-name>
    reason: <一短句——为什么合并，不只是 "相似">
prunings:
  - name: <skill-name>
    reason: <一短句——为什么 archive 且无合并目标>
```

你移到 .archive/ 的每个 skill **必须** 出现在 consolidations / prunings
两个列表的其中一个。如果你把 X 整合进 umbrella Y（patch 了 Y / 给 Y 写了
references 文件 / 创建 Y 时吸收了 X 的内容），X 属于 `consolidations`，
带 `into: Y`。如果你 archive 了 X 但没合并——确实 stale / 无关 / 过时——
X 属于 `prunings`。

9.8 Dry-run 模式

用户可以跑 hermes curator run --dry，curator 会在原 prompt 前贴一段 CURATOR_DRY_RUN_BANNER：

═══════════════════════════════════════════════════════════════
DRY-RUN — REPORT ONLY. DO NOT MUTATE THE SKILL LIBRARY.
═══════════════════════════════════════════════════════════════

This is a PREVIEW pass. Follow every instruction below EXCEPT:

  • DO NOT call skill_manage with action=patch, create, delete,
    write_file, or remove_file.
  • DO NOT call terminal to mv skill directories into .archive/.
  • DO NOT call terminal to mv, cp, rm, or rewrite any file under
    ~/.hermes/skills/.
  • skills_list and skill_view are FINE — read as much as you need.

Your output IS the deliverable. Produce the exact same human-readable
summary and structured YAML block you would produce on a live run —
but describe the actions you WOULD take, not actions you took. A
downstream reviewer will read the report and decide whether to approve
a live run with `hermes curator run` (no flag).

If you accidentally take a mutating action, say so explicitly in the
summary so the reviewer can revert it.
═══════════════════════════════════════════════════════════════
═══════════════════════════════════════════════════════════════
DRY-RUN —— 仅报告。不要 mutate（修改）skill 库。
═══════════════════════════════════════════════════════════════

这是一次 PREVIEW（预览）pass。遵守下面的每一条指令，**除了**：

  • 不要用 skill_manage 调 action=patch / create / delete /
    write_file / remove_file。
  • 不要用 terminal 把 skill 目录 mv 到 .archive/。
  • 不要用 terminal 在 ~/.hermes/skills/ 下做 mv / cp / rm / 重写
    任何文件。
  • skills_list 和 skill_view 没问题——想读多少读多少。

你的输出就是 deliverable（交付物）。产出和真跑一遍完全相同的
人类可读 summary 和结构化 YAML block —— 但描述 **你会做**
什么动作，而不是 **你做了** 什么动作。一个下游 reviewer 会读这份
报告，决定要不要批准真跑一次（用 `hermes curator run` 不带 flag）。

如果你不小心做了 mutating 动作，在 summary 里显式说明，
让 reviewer 能回滚。
═══════════════════════════════════════════════════════════════

dry-run 跑完不会 bump last_run_at 也不会 bump run_count——一次预览不应该把下次定期 curator 的时间推迟。

9.9 安全护栏总结

护栏实现
Pre-run snapshot curator_backup.snapshot_skills() 在 LLM pass 前快照整个 skills/ 目录，万一 LLM 跑歪了可以回退。即使 snapshot 失败也不阻塞 curator（最坏情况只是失去 rollback）
从不 delete _delete_skill 实际把目录 mv 到 .archive/，硬规则在 prompt 里也复述了一遍
只动 agent-created _render_candidate_list() 用 skill_usage.agent_created_report() 已经过滤过；bundled / hub-installed / user-created 都不在列表里
Pinned skip apply_automatic_transitions 里 if row.get("pinned"): continue；prompt 里也禁止 LLM 碰
Mid-run crash 不会重触发 状态在 LLM pass 之前就持久化了，crash 也不会让下一次启动立刻再 curator
独立 auxiliary 模型 通过 auxiliary.curator.{provider,model,...} 配置；不动主 session 的 prompt cache；不连主 memory plugin

9.11 用户怎么操作 Curator

命令作用
hermes curator status查看上次跑的时间、summary、报告路径、以及当前所有 pinned skill 列表
hermes curator run立刻强跑一次（绕过 idle gate）
hermes curator run --dry跑一次预览，只产报告不动 skill
hermes curator pause / resume暂停 / 恢复自动调度（写入 .curator_state.paused）
hermes curator pin <name>把某 skill 标为 pinned；调 skill_usage.set_pinned(name, True) 写到 .usage.json。pinned skill 在 §9.4 自动转换和 §9.7 LLM 整合中都被跳过
hermes curator unpin <name>取消 pinned
hermes curator archive <name>手动 archive 一个 agent-created skill（pinned 的会拒绝；要先 unpin）

设计哲学：curator 是一个 "slow loop"，相对于 per-turn review 的 "fast loop"。fast loop 负责增量写入新知识（act fast，局部），slow loop 负责定期梳理知识库结构（act slow，全局）。两个 loop 配合，让 skill 库随时间自然演化，又不会失控膨胀。
